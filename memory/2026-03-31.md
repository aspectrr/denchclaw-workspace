## AI Employee Operating Model (Refined 2026-03-31)

**Core Principle:** React → Investigate → Escalate if Needed

**Trigger Points:**
- Mentioned in Slack
- Mentioned in ClickUp
- Alert fired

**Default Behavior:**
- Act autonomously
- If it runs into an issue → ask before proceeding
- If it needs to make a change → prepare but human executes

**Hard Rule:** Any server change must be done by human (non-negotiable)

**Agent Can:**
- Investigate, query, analyze
- Prepare configs, draft PRs
- Draft communications
- Learn and refine skills per environment

**Agent Cannot (without human):**
- Run Ansible playbooks
- Restart services
- Modify configs on servers
- Deploy anything

**Skills Are Learnable:**
- Investigation skill exists at `skills/investigation/SKILL.md`
- Refines over time in each customer's environment
- Agent maintains per-customer notes

**Config Templates:**
- Agent develops its own over time based on what works
- Not imposed - learned from experience

**Technical Requirements:**
- VPS with GitHub access
- Slack + ClickUp integrations
- Network access to ES, Kafka, Logstash, client bastions

## Detailed Day-to-Day Workflows (2026-03-31)

**Source:** Collin's detailed description of actual ELK engineering work

### Workflow Categories

#### 1. Pipeline Debugging & Parsing Issues
**Description:** Debug parsing issues coming in from clients, updating configs
**Steps:**
- Receive report of parsing issue
- Check massive GitHub repo with configs and playbooks
- Identify which pipeline/config is affected
- Make manual adjustments or deploy config to dev logstash via Ansible playbook
- Iterate until parsing is correct
- Get approval from security engineers (the customers)

**Customers:** Security engineers (work in Elastic Security, set up detection rules, monitoring, CVE tracking)

#### 2. New Pipeline Development
**Description:** Set up new data pipelines from external sources
**Steps:**
- **Data Source:** External Kafka topic (SSH into client's Kafka via leap/skip bastion host)
- **If no data coming in:** Work backward through aggregator chain
- **Client Aggregators:** WireGuard instances in client infrastructure, connected to Collin's network
- **Data Flow:** Client aggregator → Proxy → Aggregator logstash/filebeat → Kafka → Dev logstash → Elasticsearch

**Development Process:**
- If pipeline already exists for this client: Apply base config files via Ansible, make targeted edits
- If new pipeline: Build from scratch (get data in, send to ES, iterate on mapping/transform)
- Approval: Security engineers review and approve

**Infrastructure:** Ansible playbooks deploy logstash configs to paths on development logstash instances

#### 3. On-CallFeed Down Investigation
**Description:** Systematic debugging when feed alerts fire
**Steps:**
1. **Alert Source:** Python script checks last timestamp vs allowed lag/downtime thresholds
2. **If over threshold:** Alert sent to call center → Slack notification → on-call engineer paged
3. **Investigation Chain (work backward):**
   - **Step 1: Kibana/ES Check** - Look at cluster, check if documents coming in, verify timeframes
   - **Step 2: Logstash Check** - Is it DLQing? Check logs for errors
   - **Step 3: Network Check** - TCP dump, are we getting data from Kafka?
   - **Step 4: Kafka Check** - Check lag, is data being sent? Is logstash up? Is Kafka up?
   - **Step 5: Aggregator Check** - If logstash aggregator: check DLQ, config issues, logs. If no data sending: check if needs restart, TCP dump
   - **Step 6: Client Escalation** - If nothing on Collin's side, reach out to client's data source

**Key Pain:** Lots of SSH hops, checking multiple systems, correlating across distributed infrastructure

#### 4. Infrastructure Work (Cluster Setup)
**Description:** Spin up new clusters or add nodes to existing clusters
**Tools:** Foreman and MAAS environments (VMs or LXD containers)
**Steps:**
- Provision VM/container in Foreman/MAAS
- SSH into host
- Configure for Elasticsearch/OpenSearch cluster role
- Getcluster running
- Set up data flow based on cluster type
- **Pain:** One-off work, don't build clusters often, but when they do it's a "massive deal"

**Environments:**
- Test environments: Safe to experiment
- Development environments: Once running, don't want to mess it up

#### 5. Utils/Scripts/Interfaces
**Description:** General infrastructure work
**Components:**
- Access management (getting people access to clusters)
- Feed ingestion (new feeds, fixing issues)
- Resource scaling (adding resources when clusters overload)
- Metrics collection (volume per member, tickets/escalations per member)
